システム・インフラ構築、デザインも引き受けます

弊社は社員二人だけの会社ですが、開発会社としては珍しく（?）エンジニアだけではなくデザイナーもいるので、システム構築もデザインも一緒に注文ができます。また、システム構築だけではなく、インフラ構築も承ります。複数社をまたいだ連絡や調整、契約を必要としないので話が早いです。

クラウドインフラ構築も一貫して。

AWS（Amazon Web Services） や GCP（Google Cloud Platform）を利用したインフラ構築もお受けしています。特に弊社ではGCPを得意としています。

必要な規模に合ったインフラ構築をデザイン・アプリ開発から一貫して行います。

デザイナーも開発チームに入って開発します。

デザイナーもエンジニアの中に入り開発チームの一員として、デザインもシステムの一部としてシステム構築と同時に開発をします。

デザインを作ってから、それをシステムに組み込むといった手順や手間が発生しないので開発速度が上がります。また、お客様は早い段階でデザインが入った形を確認することができるで、改善点、修正点の発見も速くなります。

デザイナーは動かない絵ではなく、実際にユーザーが触るものと同じシステムが動く画面を触りながら直接デザインをしていくので、その画面で起きている問題の発見や改善策が思いつくスピードも速く、精度の高いご提案ができます。

システム構築とデザイン、両方セットでのご依頼を優先します。

せっかくお声がけをいただいたお仕事の話は全部引き受けたいところなのですが、二人しかいない会社なので、すぐに売り切れてしまいます。

ですので、弊社がお客様にとって一番価値を提供できる形でお手伝いさせていただけるお仕事を優先させていただきます。

一番価値を提供できる形とは、やはり普段から一緒に様々なサービスを作ってる弊社のエンジニアの駒形と、デザイナーの町田のセットでご依頼をいただくことです。様々なサービスを一緒に作って経験を積んできた二人なので、要望や問題に対して、お互いがどのように動き、どんなツールを使い、どんな手順でどのように進めていけば、効率よくシステム構築・デザインができるかをわかっていますので、コミュニケーションに時間がかからず、その分クオリティの高いものが作れます。

一緒にサービスを育てます

お客様と私たちがチームとなって一緒にサービスを育てていきます。

お客様と同じ目標を持って開発します。

よくある、「まず仕様があり、それを作り終えるまでの工数・金額の見積りを提出し、実際に作って納品する」、というやり方はしません。

そのようなやり方には発注者、受注者が幸せになれない問題がたくさんあるのですが（詳しくはコチラ）、一番の問題は、発注者と受注者の目標が別のものなってしまうことと考えます。

受注者は見積りよりも少ない工数で仕様を満たしたものを納品することが目標になってしまい、本来持つべき目標である、システムが生み出す利益を上げることは無視されてしまいます。

そうならないために、お客様と私たちがチームになり、システムが生み出す利益を上げるために何をするべきかを一緒に考え開発をしていきます。

お客様と同じ目標を持って開発するための契約形態。

目標を達成するまでに、まずはいつまでに何をすべきかを話合い、それを実現するためには週どれくらいのペースで私たちが時間を割くかをお出しします。そのペースでチームに関わり、私たちの出す成果にご納得をしていただいている間は契約を続けていただきます。納得がいかなくなった場合はいつでも契約を解除することができます。そのときまで作っていた成果物はお客様のものとなります。

このような契約形態にすることによって、私たちは契約が長く続くように利益を上げるための成果を出し続けるので、お客様と私たちの目標が同じになります。

また、そもそもビジネスが上手くいかないことには契約は続かないので、ビジネスについて一緒に考え、開発側の立場からの優先順位付けなどのアドバイスをいたします。

ペースの調整にも柔軟に対応いたします。

プロジェクトのフェーズごとに開発に必要な開発リソースも変わってきます。立ち上げ時はガッツリ入って、リリース後はしばらくそこまでのリソースが必要がないのでペースを下げる、といったペースの調整にも柔軟に対応いたします。

弊社で提供できるリソース以上に開発者が必要になった場合は、確実にリソースを確保できる保証はないですが、もし手の空いている弊社のインターンを卒業してフリーランスになったエンジニアや、私たちが信頼している友達のエンジニア・デザイナーがいた場合はご紹介いたします。その場合、紹介料はいただきません。弊社を通しての契約にするか、直接契約にするかはご都合のいいほうをお選びいただけます。

内製化に切り替える場合も援助をいたします。

いずれ利益が大きくなって開発リソースを内製化に切り替えるときが来るかもしれません。

契約が続くように成果を出し続けてきたので、本当は契約を続けてほしいところではありますが、いいエンジニア・デザイナーが採用できるのであればビジネスの成功のためには内製化に切り替えた方がいい場合もありますので、その際は内製化に協力いたします。

開発者の採用支援、教育、開発チーム立ち上げ、開発者が働きやすいオフィス環境構築もお手伝いいたします。

FAQ

常駐ですか？リモートですか？

どちらも可能ですが、弊社のオフィスは開発がしやすい環境が整っているので、リモートでやらせていただいたほうが作業効率がいいです。

リモートといっても、作業日はチャット、ビデオチャットですぐに会話をできるようにしております。ビデオチャットを使えば会話だけなく、作業中の画面もすぐにお見せすることができるので、むしろリモートのほうがコミュニケーションが取りやすい場合もあります。

ただ、フェーズによっては、作業よりも話し合うことが多いこともありますので、その場合は貴社オフィスで作業をいたします。

すでにある開発チームに入ってもらえますか？

すでにある開発チームに入ることも可能です。ただし、開発方法があまりにも私たちのやり方と異なる場合など、価値を提供するのが難しい環境ではお断りする場合がございます。

あまり経験の多くないエンジニアで構成されたチームの場合などは、開発者としてだけでなく、開発のコンサル、メンターも兼ねてチームに加わることも可能です。

新規立ち上げでなく、既存のシステムの追加開発も依頼できますか？

既存のシステムの追加開発も承ります。

ただし、コードがひどい、アップデートがされていないなど、機能追加をする前に優先的にやるべきことをやらないと、いつそのシステムが問題を起こすかわからない、といった状態であることがたまにあります。そのような場合、機能追加よりもシステム自体の改善を優先することにご了承をいただけないと引き受けることはできません。

新規立ち上げでなく、既存のシステムのリプレイスも依頼できますか？

既存のシステムのリプレイスも承ります。

ただし、場合によってはリプレイスをしなくても、既存のシステムの改善のほうがお客様の目標を達成に近い場合もあります。リプレイスをする、しない、も含めてご相談ください。

Rails以外のシステムでも依頼できますか？

Railsにこだわらず、お客様の目標を達成するのに最適な言語、フレームワークを選択いたします。それがもしRailsでなかった場合でも対応いたします。

メンターや顧問もやってもらえますか？

どのようなエンジニアに育てたいか、どのような開発チームを作りたいか、などについて話し合い、共感をいただけましたらメンターもやらせていただきます。

まだ依頼するかも迷ってる段階なのですが、相談に乗ってもらえますか？

ぜひぜひ、ビデオチャットでよろしければ相談に乗ります。
